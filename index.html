The gap I see from where I stand

The data exists. It's just not activated.

Frame attributes — measurements, collection, material, shape, size — live on every product, but they aren't available in a consolidated location. They're stored inconsistently, not structured as typed fields, and not flowing to the systems that need them. That single gap breaks five downstream use cases simultaneously.

Store search can't filter by fit

Associates type "narrow" and get keyword results. With integer mm fields, they'd drag a lens-width slider to 44–49 and see every frame that physically fits — regardless of what the name says. Style book tags would become a sidebar filter, not a mental lookup.

Attributes scattered across systems

Collection, material, shape, and available sizes live in different places — sometimes a spreadsheet, sometimes a field that was added ad hoc. Without a canonical data model, every team maintains their own version of truth and they all drift.

AI search gets nothing to index

Google SGE and ChatGPT shopping surface products via schema.org additionalProperty structured fields. Without a clean data model publishing those fields, there's no AI search strategy — just hope. And this isn't hypothetical: customers are already asking ChatGPT to generate frame options for them. If our attributes aren't structured and published, we don't show up.

How I'd solve it

Pull once. Activate everywhere.

A CDP + tag manager layer captures structured frame attribute data from the website once and routes it to every downstream consumer — store search, Looker, schema.org, AI indexing — from a single source. The Frame Attribute Data Model v2 is the foundation that makes all of it possible. Critically, this architecture lets each team digest the KPIs most relevant to their goals: merchandising sees assortment gaps by fit tier, retail ops sees in-store availability and book coverage, e-commerce sees search conversion by attribute, and finance sees revenue by collection. One data pipeline, purpose-built outputs.

Structured frame data, pulled once from the PIM, activated across every consumer channel without per-channel engineering work.

Proof of concept

Here's my first draft prototype.

To show the concept is sound, I built a working internal frame search tool from scratch — faceted by lens width, bridge, temple, shape, and style book tag. It runs on a flat JSON data file today; the production version would connect to a live PIM with real inventory data and role-based views. Single HTML file. No framework. No build step.

  • Range sliders filter by exact millimeter — only works with integer fields in the data model
  • Style book tags stored as a multi-value array — a frame can live in multiple books simultaneously
  • Matched measurement pills turn green — built for the associate standing with a customer
  • In-store availability shown per card — inventory layer bolted on, not embedded
